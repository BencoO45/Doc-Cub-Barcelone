    1. Installation de FreePBX 17 {#installation-de-freepbx-17}

    ## 1.1 Prérequis {#1.1-prérequis}

    Pour l’installation de FreePBX 17 il nous faut une machine virtuelle avec debian 12 d’installer → tuto ici

    1.Installation de freePBX 17 :

    Grâce à un script que nous fournis Sangoma, l’installation sera facilité ( vraiment grandement )
    Il y a aura seulement 3 commandes à saisir :

    cd /tmp
    wget https://github.com/FreePBX/sng_freepbx_debian_install/raw/master/sng_freepbx_debian_install.sh -O /tmp/sng_freepbx_debian_install.sh
    bash /tmp/sng_freepbx_debian_install.sh

    Le script prend environ 30 minutes pour i

    Et concrètement que fait le script exactement ?

    Le script d’installation effectue les actions suivantes :

    1. Ajoute les dépôts Sangoma permettant de télécharger, installer et maintenir à jour FreePBX ;
    2. Installe les paquets nécessaires au fonctionnement de FreePBX ;

    Une fois l’installation terminée, il nous reste à basculer sur l’interface web de FreePBX afin de le configurer.

    ## 1.2 Configurer FreePBX ! {#1.2-configurer-freepbx-!}

    Après l’installation accéder à l’interface web depuis votre PC dans la barre de recherche :
    http://ADRESSE_IP ( c’est l’adresse IP de votre VM, dans mon cas j’ai mit http://10.33.45.140 )

    Lors de la première connexion à l’interface web de FreePBX, il sera nécessaire de renseigner un certain nombre d’informations afin de finaliser la configuration initiale.

    Nous devons notamment renseigner les éléments suivants :

    • Username : le nom d’utilisateur pour le compte administrateur. Il est différent des comptes systèmes configurés précédemment (root et le compte créé lors de l’installation de Debian) qui ne permettent que de se connecter à la console (pas à l’interface web).
    • Password : le mot de passe.
    • Notifications Email address : l’adresse e-mail associée au compte administrateur. Cette adresse e-mail sera utilisée pour recevoir les notifications de FreePBX.
    • System Identifier : le nom de notre serveur FreePBX.
    • Automatic Module Updates : permet de définir quel comportement doit adopter FreePBX lorsque des mises à jour sont disponibles. 3 choix sont possibles :
    • Enabled \= les modules sont mis à jour automatiquement à la fréquence définie juste après ;
    • Email Only \= envoi une notification par e-mail dès qu’une mise à jour d’un module est disponible ;
    • Disabled \= le système ne vérifiera pas automatiquement si des mises à jour des modules sont disponibles.
    • Send Security Emails For Unsigned Modules : envoie une notification par e-mail lorsque l’on tente d’installer un module qui n’est pas signé. Sauf si vous savez ce que vous faîtes, nous vous conseillons de laisser coché « Enabled » .
    • Check for Updates every : la fréquence de vérification des mises à jour. Dans notre exemple ci-dessous, ce sera tous les samedis entre 08h et midi.

    On a la possibilité d’activer notre installation de FreePBX. Cette activation est optionnelle, mais fortement recommandée. L’activation est nécessaire pour avoir accès à certains outils d’administration et pour installer des modules par la suite.

    Enfin, nous avons la possibilité de suivre l'assistant permettant de configurer le pare-feu intégré à FreePBX.

    Nous ne rentrons pas dans le détail ici, voici une doc complète que j’ai trouver qui vous guide dessus :

    # 2 . Migration vers freePBX 17 {#2-.-migration-vers-freepbx-17}

    ## 2.1 prérequis pour une migration. {#2.1-prérequis-pour-une-migration.}

    Le seul prérequis pour pouvoir passer de FreePBX 16 à FreePBX 17 est de disposer d’une plage de maintenance suffisante. En effet, la migration vers FreePBX17 entraînera obligatoirement une interruption de services pour les utilisateurs.

    Si l’on dispose déjà d’un serveur FreePBX 17 installé, il faut compter environ 20 à 30 minutes de temps de migration.

    Si l’on part de zéro, il faudra plutôt compter une bonne heure de travail.

    ## 2.2 Sauvegarde du système FreePBX 16 {#2.2-sauvegarde-du-système-freepbx-16}

    La toute première étape est de créer une sauvegarde complète de notre système actuel.

    Se connecter sur l’interface web d’administration de FreePBX et se rendre dans le menu Admin > Backup & Restore :

    Si l’on dispose déjà d’un protocole de sauvegarde, il suffit de le lancer en cliquant sur l’icône associée.

    Autrement, cliquez sur le bouton + Add Backup.

    Les champs à compléter sont les suivants :

    • Backup Name : le nom que l’on souhaite donner à notre sauvegarde
    • Backup Items : conserver les choix par défaut ; ainsi toute la configuration de FreePBX sera sauvegardée
    • Storage Location : choisir l’emplacement de sauvegarde ; « Local backup storage » par défaut

    Il ne nous reste plus qu’à lancer une sauvegarde en cliquant sur le bouton associé :

    Se rendre dans le menu Admin > Backup & Restore :

    Nous obtenons un message d’alerte nous prévenant des limites de la restauration :

    Finalement cliquer sur le bouton Run Restore se situant en bas à droite :

    Pour les détailles concernant ces mises en garde :
    ce message est NORMAL et ATTENDU lors d’une migration FreePBX 16 → 17.
    Notre nouvelle VM FreePBX 17 utilise Asterisk 18 ou 20 alors que chan_sip est supprimé dans ces versions, certaines extensions de l’ancienne config FreePBX 16 utilisent encore chan_sip
    Asterisk refuse donc de restaurer ces éléments.

    Cela veux dire que FreePBX 17 n’accepte plus l'extension chan_sip mais plus que pjsip

    ## 2.4 Vérifications post-migration {#2.4-vérifications-post-migration}

    La migration est terminée, tout devrait fonctionner correctement.

    Cependant, plusieurs points sont à vérifier encore..

    Redémarrer le serveur : ce n’est pas obligatoire, mais nous avons déjà remarqué que les paramètres du firewall de FreePBX étaient parfois mal chargés à l’issue d’une restauration ; un redémarrage du serveur permet de corriger le problème. De même si les ports SIP ont été personnalisés, le redémarrage permettra de prendre en compte ce changement.

    Vérifier que les comptes SIP sont bien enregistrés : il suffit de se rendre dans la menu Reports > Asterisk Info afin de voir si tous les comptes SIP remontent correctement.
